TOP 5 BOTTLENECKS (By Impact & Severity)
================================================================================

1. CALLBACK HELL - NESTED ASYNC OPERATIONS
   File: server/server.js (lines 72-166)
   Severity: HIGH
   Impact: 5-10 seconds lost
   Issue: Deep callback nesting forces sequential execution of database ops
   Fix Priority: P1 (High effort, high impact)
   
   Status: Server uses callback pattern instead of async/await
   Result: Database operations execute one-by-one instead of in parallel
   
   Example:
     db.get() -> db.run() -> db.all() -> fetch(OpenAI) -> db.run()
     Each waits for previous to finish


2. N+1 DATABASE QUERY PATTERN
   File: server/server.js (lines 81-143)
   Severity: HIGH
   Impact: 2-5 seconds lost
   Issue: 4 separate DB queries per chat message (should be 1-2)
   
   Query sequence:
     1. SELECT 1 FROM sessions WHERE session_id = ?  (line 81)
     2. INSERT INTO messages (user message)          (line 90)
     3. SELECT * FROM messages (fetch all history)   (line 98)
     4. INSERT INTO messages (AI response)           (line 142)
   
   Fix: Use UPSERT to eliminate check, limit history, batch operations


3. UNBOUNDED CONVERSATION HISTORY LOAD
   File: server/server.js (line 98)
   Severity: MEDIUM
   Impact: 1-3 seconds (grows with conversation size)
   Issue: Fetches ALL messages, no limit. Loads entire history into memory.
   
   Current SQL: SELECT role, content FROM messages WHERE session_id = ?
   Problem: No LIMIT clause, so:
     - 100-message conversation = load 100 messages
     - Wasteful since OpenAI has token limit anyway
     - SQLite table scan can be slow
   
   Fix: Add LIMIT 50, paginate if needed


4. MISSING DATABASE INDEXES
   File: server/database.js (lines 42-44)
   Severity: MEDIUM
   Impact: 0.5-2 seconds per query (scales with data)
   Issue: Only 1 index exists, but queries need 3+
   
   Current indexes: idx_session_timestamp
   Missing indexes:
     - idx_messages_session (on messages.session_id)
     - idx_messages_session_role (composite)
   
   Impact: Without indexes, SQLite does full table scans
   Problem worsens: As database grows (1K -> 10K+ messages)


5. OPENAI API SEQUENTIAL CALL (External Factor)
   File: server/server.js (lines 116-129)
   Severity: HIGH (but external)
   Impact: 5-20 seconds (DOMINANT latency source)
   Issue: No streaming support. Response blocks until fully complete.
   
   Current: Waits for entire response before returning to client
   Better: Stream tokens in real-time using SSE format
   Frontend ready: src/modules/home/views/IndexPage.tsx already
                   has streaming parser (lines 622-693)

PRIORITY 1 BOTTLENECK: Callback Hell
================================================================================

File: /Users/tiko/Desktop/knowhowai-ui-changes/server/server.js
Lines: 72-166
Severity: HIGH
Impact: 5-10 seconds latency
Issue: Nested callbacks force sequential execution

Current Flow (SERIALIZED - Slow):
┌─ db.get(check session)
   ├─ Wait 1-2ms
   └─ db.run(save user message)
      ├─ Wait 1-2ms
      └─ db.all(fetch history)
         ├─ Wait 2-5ms
         └─ fetch(OpenAI API)
            ├─ Wait 5-20 seconds (!)
            └─ db.run(save AI response)
               └─ res.json() -> response sent

Better Flow (PARALLEL - Fast):
┌─ db.run(save user message) ─┐
│                             ├─ Wait in parallel: 2-5ms
└─ db.all(fetch history) ─────┘
   └─ fetch(OpenAI API)
      ├─ Wait 5-20 seconds
      └─ db.run(save AI response) in background
         └─ res.json() -> response sent immediately

Savings: Each callback level adds ~1-2ms overhead × 5 levels = 5-10 seconds

Fix: Convert to async/await with Promise.all() for parallel operations
